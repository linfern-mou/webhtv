# 纯音频播放界面美化设计方案

分支：`design/audio-player-ui-polish-20260702`

目标：当播放内容被识别为纯音频、音乐类视频、带歌词音频或 K 歌伴奏时，播放器不再呈现“黑色视频窗口 + 影视详情页”的观感，而是切换为更接近成熟音乐播放器的 `Now Playing` 体验。手机端和电视端都需要独立设计，电视端不能只是放大手机界面。

## 结论摘要

优先做一个“音频舞台 Audio Stage”模式：复用现有播放内核、歌词、K 歌、队列和站点数据，只替换纯音频状态下的视觉层级与布局。第一阶段不碰解码链路，重点解决用户第一眼观感、歌词可读性、队列可达性、遥控器焦点和音频状态展示。

核心改造建议：

1. 手机端：把顶部 220dp 黑色视频区域替换为沉浸式音频舞台，使用封面/海报/站点图做背景，中央呈现封面、歌名、来源、音频状态、进度、控制和歌词。
2. 电视端：把 400x225 黑色小窗替换为全屏 `Now Playing`，左侧大封面，右侧歌词/队列/信息面板，底部遥控器控制轨。
3. 歌词：纯音频下歌词应该是主内容，不应继续作为视频画面上的小 overlay；K 歌状态条也要升级为独立区域。
4. 音效能力：先做状态可视化入口，如格式、采样率、声道、音频直通、均衡器、K 歌、歌词、LUT 不在音频页露出。硬核音频处理能力以后分阶段接入。
5. 兼容策略：仅在 `audioOnly` 或 `isAudioContent/isMusicLike` 成立时启用新布局，视频播放路径保持现状。

## WebHomeTV 当前状态

已具备的基础能力：

1. `PlaybackActivity` 已有 `audioOnly` 状态。
2. 手机/电视 `VideoActivity` 已经接入 `LyricsController.isAudioOnly(player())`、`isAudioContent`、`isMusicLike` 等判断。
3. 已有 `LyricsOverlayView`，支持多行歌词和逐字 karaoke 高亮。
4. 已有 `KaraokeStatusView` 和 `KaraokeResultView`，具备 K 歌状态、打分、音高时间线基础。
5. 手机与电视现有布局都复用 `activity_video.xml`，控制、进度、OSD、LUT 面板都挂在视频容器内。

主要问题：

1. 手机端纯音频仍占用固定 220dp 黑色视频框，歌词被限制在小窗口内，视觉上不像音乐播放器。
2. 电视端纯音频仍是 400x225 视频小窗，右侧是影视详情信息，客厅场景缺少大封面、大歌词和遥控器控制轨。
3. 音频元数据没有被视觉化，用户看不到格式、采样率、声道、音频直通、歌词/K 歌状态。
4. 歌词、队列、选集、详情的优先级仍按视频内容组织，不适合音乐/音频内容。
5. K 歌能力已有技术基础，但 UI 像“挂件”，没有成为音频场景的一等功能。

## 设计原则

1. 纯音频必须从“视频页的空画面”转为“音乐播放器的正在播放页”。
2. 手机端优先拇指操作和滑动切换，电视端优先遥控器焦点和 10-foot 可读性。
3. 信息层级固定：封面/标题 > 进度/控制 > 歌词 > 队列 > 来源/技术信息。
4. 背景可以精致，但不能影响歌词和按钮可读性；必须有暗色遮罩和动态取色兜底。
5. 现有视频播放功能不被干扰，纯音频模式只在明确命中时启用。
6. 硬核音频能力先展示状态，避免第一阶段引入复杂 DSP 影响稳定性。

## 手机端界面方案

### 结构

手机端建议新增纯音频状态下的 `Audio Now Playing` 布局，替代当前顶部视频区域。页面结构：

1. 背景层：使用 `ContextWallImageView` 或封面/海报做全屏背景，模糊 + 暗色遮罩 + 底部渐变。
2. 顶部栏：返回、站点/来源、更多；标题不抢主视觉。
3. 封面区：居中正方形封面，宽度约屏宽 64%-72%，最大不超过 320dp。无封面时用来源色生成音乐占位图，可带轻量频谱线条。
4. 元信息区：主标题、子标题、站点/分组、当前集/文件名。长文本最多两行，避免横向滚动。
5. 音频 badge：格式、码率/采样率、声道、歌词、K 歌、缓存/网盘来源，控制在一行横向可滚动小标签。
6. 进度区：进度条、已播、剩余/总时长、缓冲状态。拖动行为沿用现有播放器。
7. 主控制区：上一首/后退、播放暂停、下一首/快进，图标尺寸比视频页更大。
8. 二级控制区：歌词、队列、音效、倍速/定时、收藏/缓存。不要把 LUT 放到音频首页。
9. 歌词区：默认展示当前歌词 + 前后各 1-2 行；向上滑可进入全屏歌词，向下回到封面。
10. 队列区：底部抽屉展示当前站点/播放列表/选集，支持切换、删除、排序和当前播放定位。

### 推荐布局状态

| 状态 | 页面重点 | 说明 |
| --- | --- | --- |
| 无歌词 | 封面 + 进度 + 队列 | 封面更大，歌词区显示“暂无歌词”但不占大面积 |
| 普通歌词 | 歌词作为下半屏主内容 | 当前行 20-24sp，高亮色使用暖黄色或从封面取亮色 |
| 逐字歌词 | 当前行逐字高亮 | 复用 `LyricsOverlayView` 的 karaoke span，但字号和行距按音乐页重设 |
| K 歌 | 歌词 + 音高/得分 | `KaraokeStatusView` 不再放右上角，改为歌词下方或进度上方的横向状态条 |
| 网络缓冲 | 背景保持，封面不闪 | 中央不弹大 loading，进度条上方显示轻量状态 |
| 横屏手机/平板 | 左封面右歌词 | 参考电视布局缩小版，避免横屏仍垂直堆叠 |

### 视觉规格建议

1. 背景：封面模糊半径 20-32dp，黑色遮罩 55%-70%，底部加深，保证白字可读。
2. 封面圆角：8dp，符合现有设计约束，不做夸张大圆角。
3. 主标题：20-24sp，最多两行；副标题 13-15sp。
4. 当前歌词：20-24sp；前后歌词 14-16sp，透明度 55%-75%。
5. 主播放按钮：56-64dp，上一首/下一首 44-48dp。
6. 颜色：避免单一紫蓝或深蓝主题，优先封面取色 + 黑白灰 + 一种高亮色。无封面时使用中性深灰、墨绿、琥珀、蓝灰混合，不做大片渐变球。
7. 动效：封面淡入、歌词滚动、控制区显隐即可；不做复杂旋转唱片作为默认效果，防止廉价和耗电。

## 电视端界面方案

电视端应作为独立 10-foot 播放界面，而不是手机界面放大。建议纯音频播放时进入全屏 `Audio Leanback`：

### 默认布局

1. 全屏背景：封面/海报模糊铺底，叠加深色遮罩，四周留电视安全边距。
2. 左侧视觉区：大封面，建议 360-460dp 正方形，下面展示来源、格式、声道 badge。
3. 右侧内容区：标题、歌手/站点/分组、当前歌词、下一句歌词。歌词区域必须比影视简介优先。
4. 底部控制轨：上一首、后退 10s、播放/暂停、前进 10s、下一首、歌词、队列、音效、更多。所有按钮有清晰焦点态。
5. 右侧/底部队列：按遥控器 `菜单/下` 可打开队列抽屉，列表行高 56-72dp，当前播放行高亮。
6. 顶部状态：时间、播放源、网络/缓存状态可以很轻，不压主内容。

### 焦点设计

默认焦点：播放/暂停。

焦点顺序：

1. 左右：上一首、后退、播放、前进、下一首、歌词、队列、音效、更多。
2. 上：进入歌词/封面区域，OK 切换歌词全屏或封面信息。
3. 下：打开队列/选集列表。
4. 返回：优先关闭队列/面板，再退出播放。
5. 长按 OK：可打开音频详情或当前文件操作。

遥控器原则：

1. 所有焦点按钮宽高不小于 56dp。
2. 焦点态要有边框/放大/亮度变化，不只靠颜色。
3. 歌词全屏状态下，左右仍能切歌，上下可调歌词偏移或切换歌词行浏览。
4. 不显示需要鼠标悬停才能理解的控件。

### 电视歌词与 K 歌

普通歌词：

1. 默认右侧显示 3-5 行，当前行 32-40sp。
2. 无歌词时显示封面和队列，不要空出黑框。
3. 歌词可一键全屏，全屏时中央大字，底部保留半透明控制轨。

K 歌模式：

1. 中央显示当前歌词，下一句在下方。
2. 底部显示音高轨、当前分数、连击/稳定度。
3. 右上角小状态只保留麦克风权限/错误提示，不展示完整得分。
4. 电视端 K 歌应预留“麦克风延迟”和“难度”快速入口。

## 音频硬核能力入口设计

第一阶段只做状态展示和入口，不建议立即引入复杂音频处理。可展示：

1. 源格式：MP3、AAC、FLAC、ALAC、APE、WAV、DTS、AC3、EAC3 等。
2. 采样率/位深：44.1kHz、48kHz、96kHz、24bit 等，能拿到就显示，拿不到不乱猜。
3. 声道：2.0、5.1、7.1、Atmos、DTS-HD 等。
4. 输出状态：音频直通开/关、系统解码、软解、蓝牙输出、HDMI/电视扬声器。
5. 歌词状态：无歌词、普通 LRC、逐字歌词、K 歌音高。
6. 网络状态：直连、代理、本地缓存、网盘来源。

后续可评估的能力：

1. 均衡器/预设/AutoEQ。
2. 重放增益、响度均衡、夜间模式。
3. Crossfeed，把耳机听感做得更自然。
4. 声道混音：单声道、人声增强、中央声道增强。
5. 可视化：频谱、波形、VU 表。电视端可以作为屏保式视觉，但不应影响性能。
6. 音频链路解释：源文件 -> 解码 -> DSP -> 输出设备，参考 Roon 的透明信号路径。
7. 睡眠定时和播放完当前曲目停止。
8. 队列编辑、多队列或临时播放列表。

## WebHomeTV 借鉴清单

优先借鉴：

1. Apple Music/Spotify 的清晰 `Now Playing` 层级。
2. Plexamp 的封面取色、背景氛围和精致但克制的音乐感。
3. Musicolet 的多队列/文件夹思路，用于网盘和站点合集。
4. Roon 的音频链路透明展示，作为未来硬核能力入口。
5. Poweramp/UAPP 的音质 badge 和音效入口，但不要照搬复杂参数页。
6. Kodi/foobar2000 的电视形态支持和遥控器优先交互。

暂不建议借鉴：

1. 默认旋转唱片、花哨频谱、过多动效。容易廉价且影响电视性能。
2. 复杂 DSP 参数直接暴露在播放首页。用户会被打扰，且实现成本高。
3. 流媒体社交玩法，如年度回顾、好友合听。与 WebHomeTV 当前核心场景关系不大。
4. 过度依赖在线元数据。站点/网盘资源命名不稳定，先做好本地/已有信息兜底。

## 实施拆分

### 阶段一：纯音频视觉分流

1. 增加纯音频布局容器，手机和 leanback 分别实现。
2. 命中 `audioOnly/isAudioContent/isMusicLike` 时显示 audio stage，隐藏黑色视频背景。
3. 复用现有播放控制、进度、歌词更新和 K 歌状态数据。
4. 音频态隐藏 LUT 快捷面板等视频专用入口。
5. 增加封面/海报/站点图/默认占位图的统一选择策略。

验收标准：

1. 手机端纯音频第一屏不再出现黑色视频块。
2. 电视端纯音频不再是 400x225 小窗。
3. 播放、暂停、拖动、切集、返回行为与现有视频页一致。
4. 无歌词、有歌词、K 歌三种状态都不空、不挤、不遮挡。

### 阶段二：歌词与队列强化

1. 纯音频歌词使用专用字号、行距、滚动策略。
2. 手机端增加歌词全屏/封面视图切换。
3. 电视端增加队列抽屉和全屏歌词模式。
4. 当前播放项在队列中自动定位，高亮显示。
5. K 歌状态条适配手机/电视布局。

验收标准：

1. 手机竖屏歌词不被 220dp 容器限制。
2. 电视 3 米观看距离能读清歌词。
3. 遥控器可以无障碍打开/关闭队列并切换曲目。

### 阶段三：音频状态与高级入口

1. 展示格式、采样率、声道、直通、歌词/K 歌状态 badge。
2. 增加“音效”入口，先承载现有声音相关设置。
3. 预留未来 EQ、重放增益、声道增强、音频链路解释页面。
4. 增加睡眠定时/播放完停止等音乐场景操作。

验收标准：

1. 能拿到的音频信息准确显示，拿不到的信息不显示占位假数据。
2. 音效入口不会影响视频播放页。
3. 电视端音频 badge 可读且不抢歌词焦点。

## 风险与注意点

1. 识别风险：`isMusicLike` 可能误判音乐类视频，建议首版只对 `audioOnly` 和明确音频 MIME/扩展启用完整 audio stage，音乐类视频可采用轻量模式。
2. 封面来源不稳定：站点数据可能没有图，必须有默认占位策略，不能黑屏。
3. 歌词性能：逐字高亮 50ms 刷新在大屏和低端设备上要关注过度 invalidate。
4. 电视焦点：新增布局后必须逐项设置 `nextFocus`，否则遥控器会丢焦。
5. K 歌与音频页冲突：K 歌状态不应遮挡歌词，也不应让控制轨跳动。
6. 横竖屏切换：手机横屏、平板、折叠屏需要响应式布局，不要只按竖屏写死。
7. 视频能力隔离：LUT、字幕、画质等视频入口在音频模式不应混入首页，但仍可在更多里保留必要设置。
